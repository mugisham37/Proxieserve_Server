## Section Two: The Architecture You Are Building Into

This backend follows a strict layered architecture and you must understand it as a system, not just as individual files.

Every domain feature lives in its own folder under app/modules/. A module folder always contains at minimum five files: models.py, schemas.py, repository.py, service.py, and router.py. Some modules also have constants.py (for module-level string constants), events.py (for domain events published to the in-process event bus), or jobs.py (for ARQ background job functions specific to this module). You will be creating five new module folders: services, applications, documents, messages, and assignments.

The models.py file defines SQLAlchemy ORM classes. It imports the declarative base or a shared base class used by the auth module. Every model uses string primary keys with a consistent prefix (for example, all user IDs start with "usr_", all refresh tokens start with "rti_"). Every model has created_at and most have updated_at timestamps. Foreign keys use String(64) type matching the primary key format. Relationships between models use SQLAlchemy's relationship() with lazy="raise" (or equivalent) so that accessing a relationship without eagerly loading it raises an error rather than silently triggering a lazy query that would fail in async context.

The schemas.py file defines Pydantic v2 models for request bodies and response payloads. Request schemas validate and parse incoming JSON. Response schemas define what data is returned. It is common to have separate schemas for creating a resource (CreateServiceRequest), updating it (UpdateServiceRequest), and returning it (ServiceResponse, ServiceDetailResponse). The schemas use Field for validation constraints and validators for custom logic.

The repository.py file is the only place where SQLAlchemy queries appear. Every method receives an AsyncSession as its first argument. Methods that read data use select() with optional where(), order_by(), limit(), offset() clauses, and selectinload() or joinedload() options for related data. Methods that write data call session.add(), session.flush() (to get the generated ID or check constraints without committing), and leave the commit to the calling service. Methods that update data fetch the row first, modify attributes, then flush. The repository never commits — that responsibility belongs to the service layer.

The service.py file contains the business logic. It receives an AsyncSession and any other dependencies (like the job queue manager, the email notifier, the current user's token data) as constructor arguments. It calls repository methods, applies validation and business rules, calls the ARQ job queue to enqueue background jobs, publishes domain events if needed, and then calls await session.commit() after all operations are complete. If any operation fails, the service raises an AppError subclass with the appropriate HTTP status code. The service layer is where ownership checks happen — if a client tries to access an application that belongs to a different client, the service raises a ForbiddenError.

The router.py file is where FastAPI route handlers live. Route handlers declare their dependencies using Depends() — typically the async session, the current user (from one of the REQUIRE_* dependencies), a rate limiter, and the service class. The route handler calls the service method and returns success_response(message="...", data=response_schema.model_validate(result)). It does not contain any SQL, any business logic, or any file I/O. Its only job is to receive the HTTP request, call the service, and return the HTTP response in the standard envelope format.

The core/api.py file defines the ApiResponse envelope and the success_response() and error_response() helper functions. Every route handler must return a success_response(). Every error is thrown as an AppError subclass which is caught by the global exception handler in middleware.py and automatically converted to an error_response(). You must never return a raw dict or a non-enveloped response from any endpoint.

The core/ratelimit.py defines the rate_limit dependency factory. To add rate limiting to a route, you add a parameter to the route handler's signature with Depends(rate_limit("name-of-this-route", limit=20, window_seconds=60)). The name must be unique across the entire application. Apply rate limiting to every new endpoint you create, using limits proportional to how frequently that endpoint might legitimately be called. High-volume read endpoints can have higher limits. Write operations and uploads should be more restrictive.


## Section Three: Role Hierarchy and Power Dynamics

There are exactly three roles in this system: staff:admin, staff:agent, and client. Every route in the new modules you build must be protected by exactly one of the four role dependency constants: REQUIRE_ADMIN, REQUIRE_AGENT, REQUIRE_AGENT_OR_ADMIN, or REQUIRE_CLIENT. No endpoint in this module is truly public except for the service catalogue read endpoints and the application tracker endpoint, which require no authentication at all.

The admin (staff:admin) is the absolute authority over the application process structure. The admin creates service templates — the admin is literally the one who decides what every application process looks like, what questions are asked, what documents are required, how many steps there are, and what the pricing tiers are. There is no part of a service template that a client or agent can create, modify, or influence. The admin also has full visibility into every application submitted by any client to any agent. The admin can assign any unassigned application to any agent, can reassign an application from one agent to another at any time, and can force any application to any status regardless of the normal state machine rules. The admin sees the unread messages of all applications. The admin can see all internal agent notes. If you are ever in doubt about whether the admin should be able to do something, the answer is yes.

The agent (staff:agent) is the operational processor. An agent is a staff member created by the admin. The most important constraint on the agent is that they can only see and act on applications that are assigned to them. An agent must never be able to access an application assigned to a different agent, nor can they see unassigned applications except through the unassigned queue endpoint that allows them to claim one. Within their assigned cases, the agent has significant operational authority: they update the application status (within the rules of the state machine), they upload documents on behalf of the applicant when needed, they review and mark QC on uploaded documents, they send messages to the client, they write internal notes visible only to staff, and they complete or reject applications. The agent cannot create or modify service templates. The agent cannot reassign an application to another agent — only the admin can do that.

The client is the end user. The most important constraint on the client is that they can only see and interact with their own applications. A client must never be able to access another client's application data, documents, or messages. Within their own applications, the client fills out the wizard, uploads required documents, sends messages to their assigned agent, reads messages from the agent, and tracks the status of their application. The client can cancel an application, but only when it is in the received or awaiting_client status — not once an agent has begun active processing. The client has zero control over the process structure. The wizard steps, the questions, the document requirements, the pricing — all of this is defined by the admin and delivered to the client as a fixed configuration that they must follow.

The key architectural insight that must guide every decision you make is this: the service template is the blank canvas that the admin fills in, and what the admin fills in is exactly and only what the client will see and be asked to do. If the admin creates a passport renewal service with two document requirements and three form fields, then when a client applies for passport renewal they will be shown exactly those two document upload slots and exactly those three form fields — no more, no fewer, no different. The backend enforces this by reading the service template at submission time and validating the client's submitted data against the template's requirements.


## Section Four: What the Admin's Service Template System Looks Like

When you implement the services module, you are building the system that gives the admin a blank canvas and the tools to fill it in. The admin's service template has five interconnected layers that you will store across five related database tables.

The first layer is the core service record. It contains the unique slug that identifies the service in URLs (like "passport-renewal" or "business-registration"), the display name, the category (one of: identity, business, tax, welfare, permits, other), a full prose description, a short one-line description for cards and lists, a hex color string used to color-code the service in the UI, an icon name string, a status field that controls visibility (draft means not visible to the public, active means visible and accepting applications, paused means visible but not accepting new applications, unavailable means temporarily hidden, archived means permanently retired), a version integer that increments when the admin makes significant changes, a flag for whether this service is featured on the homepage, and a reference to which admin user created it. When the admin creates a new service, it always starts in draft status. The admin must explicitly publish it to active status before clients can see or apply for it.

The second layer is the process steps. These are an ordered list of milestone names defined by the admin that describe the human-readable journey the application will go through. For example, a passport renewal service might have steps called "Identity Verification", "Document Review", "Authority Submission", "Processing", and "Collection Ready". These steps are what the client sees in the progress tracker on the frontend — their application is at step 2 of 5, for example. These steps are distinct from the application's internal technical statuses (which you will implement in the applications module). The steps are purely for client communication. Each step has a step number, a title, and an optional description. The admin replaces the entire list of steps whenever they update it — you do not add or remove individual steps, you accept a new ordered array and replace the current steps for that service.

The third layer is the document requirements. These are the specific documents the admin requires applicants to upload. Each requirement has a machine-readable key unique within this service (like "national_id", "passport_photo", "company_registration_certificate"), a human-readable label for display, a longer description explaining what the document should show, a document type from the enumeration (id, certificate, photo, form, proof, other), a flag for whether it is required or optional, the maximum file size in megabytes, and a list of allowed MIME types stored as a JSON array. The order of requirements in the upload step is controlled by the sort_order field. When the client reaches step three of the wizard, this table is what determines exactly which upload slots appear. When the server validates an uploaded file, it checks the MIME type against the allowed_mime_types for that requirement key. Like steps, the admin replaces the entire list of document requirements when updating.

The fourth layer is the form fields. These are the dynamic questions that appear in step two of the wizard — the service-specific information that goes beyond the personal details collected in step one. The admin defines each field with a machine-readable key (like "application_type", "company_name", "number_of_directors"), a label, a field type from the enumeration (text, textarea, select, radio_card, date, switch, checkbox), optional help text, whether it is required, an optional array of choice options for select and radio_card fields (each option having a value string and a label string stored as a JSON array of objects), optional conditional display logic (a field key that must have a specific value for this field to appear, enabling fields that only show up when the user makes a certain selection), a sort order, a maximum character length for text fields, and a placeholder string. The admin replaces the entire list of form fields when updating. At submission time, the server reads all non-conditional required fields and all conditional fields whose condition is satisfied, then validates that the submitted service_data from the client contains all of them.

The fifth layer is the pricing tiers. There are exactly three tiers per service: standard, express, and urgent. Each tier has a display name, a description, the platform fee in the smallest currency unit (Rwandan francs), a separate government fee in the same unit, the estimated turnaround time in business days, a JSON array of feature strings (like "Priority processing", "Same-day acknowledgement", etc.), and a flag for whether that tier is currently available. The admin can update individual tier pricing and toggle tier availability without needing to replace all tiers. When a client submits an application, they choose one of the available tiers, and the chosen tier's fee and ETA are snapshotted onto the application record.


## Section Five: The Application Lifecycle and Status Machine

An application is born when a client submits the five-step wizard. The submission creates a record with an auto-generated PRX code, sets the status to received, and stores the client's personal information (from step one) and service-specific responses (from step two) as JSON fields on the application record. Documents are uploaded separately after the application is created and linked via the application code. The application record also snapshots the service's name and slug at submission time so that even if the service is later renamed or archived, the application always knows which service it was for.

The PRX code is a unique, human-readable identifier for the application. It is formatted as PRX followed by the eight-digit date in YYYYMMDD format, a hyphen, and then five uppercase characters chosen from an unambiguous alphabet that excludes characters that look like each other (no letter I, letter O, digit 1, or digit 0). The server generates this code at submission time, checks for a collision in the database (extremely unlikely but must be handled), and retries if a collision occurs. The code must be stored in a unique-indexed column.

Once the application is in received status, the admin assigns it to an agent. The admin selects any active agent and calls the assign endpoint. This sets the assigned_agent_id and assigned_at fields on the application, creates an entry in the assignment history table, and triggers a status transition from received to under_review. A system message is created in the application's conversation saying that the application is now under review, and the client receives an email notification.

The complete set of valid status transitions, who can trigger them, and what must happen when they do is as follows. The received status can transition to under_review (this happens automatically when the admin assigns an agent or when an agent claims the application from the unassigned queue — not a separate manual transition), or it can transition to cancelled (triggered by the client if they choose to cancel early, or by the admin). The under_review status can transition to in_progress (triggered by the agent when they start active work), to awaiting_client (triggered by the agent when they need something from the client), or to rejected (triggered by the agent or admin if the application is determined to be invalid or ineligible at this stage). The in_progress status can transition to submitted_to_authority (triggered by the agent when they have formally submitted all materials to the external authority), to awaiting_client (triggered by the agent if they discover mid-processing that they need something from the client), or to rejected (triggered by the agent or admin). The awaiting_client status can transition to in_progress (triggered by the agent when the client has provided what was needed, or triggered automatically when the client uploads a new document in response to the request), or to rejected (triggered by the agent or admin if the client does not respond in time or if what they provided is insufficient), or to cancelled (triggered by the client if they decide to abandon the application). The submitted_to_authority status can transition to awaiting_response (triggered by the agent to indicate they are now waiting on the external authority), or directly to completed (if the authority responds immediately), or to rejected (if the authority rejects it). The awaiting_response status can transition to completed (triggered by the agent when the authority approves), to rejected (triggered by the agent when the authority rejects), or to in_progress (triggered by the agent if they need to go back and resubmit something). The completed and rejected and cancelled statuses are terminal — no transitions out of them exist, except that the admin can force a status change out of any status including these.

Every status transition must be enforced in the service layer. The service method that handles a status change must first look up the current status of the application and compare the requested new status against a dictionary of valid transitions. If the transition is not valid, it must raise a new exception type called InvalidStatusTransitionError that extends AppError with HTTP 422 status code. The error message should indicate the current status and which statuses it can validly transition to. This validation must happen in the service layer, not in the router and not in the repository.

Every status transition, without exception, must result in three things: first, the application record's status field is updated in the database; second, a new row is inserted into the application_status_history table recording the new status, who made the change, what their role was, an optional note, and the timestamp; third, a system message is created in the application's conversation thread via the messages module's service layer (you will call the message service from within the application service when creating system messages). In addition to these three mandatory things, most transitions also trigger an email notification to either the client or the agent — the exact triggers are described in the notifications section later in this document.


## Section Six: The Document Upload and Storage Architecture

File handling is an area where many implementations go wrong and you must follow the prescribed approach precisely. The golden rules are: never buffer an entire upload in memory, always validate the actual file type by inspecting its bytes rather than trusting the Content-Type header, use async file I/O for all disk operations, run quality checking in a background job not synchronously, and always enforce access control before streaming any file to the requester.

The file storage system must be built as an abstraction behind a storage backend interface. Define a base class in app/core/storage.py with two async methods: one to save a file (accepting a file stream and a destination path, returning the full stored path), and one to retrieve a file (accepting a stored path, returning a stream or the bytes). Implement a local filesystem backend as the concrete class for now. The local backend uses an UPLOAD_DIR environment variable (new, must be added to the Settings config class and to the .env.example file) as the root directory. Within that root, files are stored in the path: applications/the_application_id/documents/the_document_id/original_filename. This path structure means you can locate all files for an application, you can locate the specific file for a document record, and the original filename is preserved for download. Create the directory structure with os.makedirs or pathlib.Path.mkdir(parents=True, exist_ok=True) before writing.

For the upload endpoint, the route handler must accept a multipart form request (not JSON). The route parameters are: the application code as a path parameter, the requirement key as a Form field (a string identifying which document slot this file fills), and the actual file as an UploadFile parameter. The route handler passes these to the document service. The document service is responsible for all validation and storage.

In the document service, the first thing to do is look up the application by code and verify that the requesting user has permission to upload to it (the client must own it, an agent must be assigned to it, or the user must be admin). Then look up the service's document requirements to find the requirement matching the provided requirement_key and read its constraints (max file size and allowed MIME types). Then begin streaming the file from the UploadFile object in chunks. As you read chunks, keep a running total of the bytes read — if the total exceeds the requirement's max_size_mb converted to bytes, stop reading, delete any partial file you may have written, and raise a FileTooLargeError. Write each chunk to a temporary file first using aiofiles. After the upload is complete, read the first 128 bytes of the file using aiofiles and pass them to the filetype library's guess() function or to python-magic's from_buffer() function to detect the actual MIME type. If the detected MIME type is not in the requirement's allowed_mime_types list, delete the temporary file and raise a DocumentTypeNotAllowedError. If validation passes, move the temporary file to its permanent storage path. Then create the document record in the database with the storage path, MIME type, file size, uploader info, requirement key, version number (starting at 1), QC status set to pending, and is_active set to true. Finally, enqueue an ARQ background job to perform quality checking on this document. Return the created document record data to the router.

If a document already exists for this application and this requirement_key and it is the current active version (is_active is true), then when a new upload comes in for the same key, you do not delete the old record. Instead, you set the old record's is_active to false and set its replaced_by field to the ID of the new document record. You set the new document record's version to one more than the previous record's version. This gives a full replacement history.

The quality checking background job is registered in the ARQ worker alongside the existing email and SMS jobs. The job receives the document_id as its argument. It opens a fresh database session (because the job runs in a separate process), fetches the document record by ID, reads the file from storage using aiofiles, and then runs quality checks. For image files (JPEG, PNG, WEBP), use Pillow to open the image, check that its width and height are both at least 400 pixels, check that the image is not nearly uniform in color (a completely white or black image is likely a scanning error), and check that the image is not extremely blurry by computing a Laplacian variance on a grayscale version of the image and checking it against a minimum threshold. Set the qc_status field to pass if all checks succeed, to warn if checks detect possible issues but the document might still be usable, or to fail if the image is clearly unsuitable. Update the qc_notes field with a JSON object describing what each check found. If the qc_status is warn or fail, enqueue an email notification to the client asking them to check their document.

For the document download endpoint, accept the document_id as a path parameter, look up the document record, verify access (the application's client, the assigned agent, or any admin), fetch the file from storage, and return a StreamingResponse. Set the Content-Type header to the stored MIME type. For downloads (the caller wants to save the file), set Content-Disposition to attachment with the original_filename. For previews (the caller wants to view it inline), set Content-Disposition to inline. Include the Content-Length header from the stored file_size_bytes. Stream the file in 64-kilobyte chunks using a generator function.


## Section Seven: The Messaging and Conversation System

Every application has a conversation thread containing all messages between the client and the agent. The messages table stores all of them in a single table with flags to distinguish their type and visibility.

A message has three types. A public message is a normal message sent by either the client or the agent that both parties can see. An internal note is a message sent by an agent or admin that is marked with is_internal equal to true and is never shown to the client under any circumstances. A system message is automatically generated by the application service layer when a status transition occurs, marked with is_system equal to true, and shown to the client as a neutral informational announcement (not attributed to any human sender). The message record stores the sender's user_id, the sender's role at the time of sending, the text content, the is_internal flag, the is_system flag, a JSON array of document IDs for any referenced attachments, a boolean for whether the client has read this message, a timestamp for when an agent first read a message sent by the client, and the creation timestamp.

When the client fetches the conversation for their application, the repository query must filter out all rows where is_internal is true. The client must never see internal notes under any circumstances — not in the response envelope, not as a count, not in any summary. When the agent or admin fetches the conversation, they receive all messages including internal notes.

When the client posts a message, the service creates a new message record with is_internal set to false and is_system set to false, and then enqueues an email notification job to alert the assigned agent. When an agent posts a message, the service creates the record with is_internal set to whatever the request body says (true for an internal note, false for a message to the client), and if is_internal is false, enqueues an email notification to the client.

System messages are created by calling an internal method on the messages service from within the application service — not via an HTTP endpoint. This internal method creates a message with sender_id set to null, sender_role set to "system", is_internal set to false, and is_system set to true. The content is a fixed string appropriate to the transition (for example, when status changes to under_review: "Your application has been assigned to an agent and is now under review." When it changes to awaiting_client: "Your agent needs additional information from you. Please check the message below for details." When it changes to completed: "Congratulations — your application has been successfully completed.").

The messages endpoints use rate limiting. Clients can send at most 30 messages per minute. Agents have a slightly higher limit. The read endpoints can have more generous limits.


## Section Eight: The Assignment System

Applications are assigned to agents either by the admin explicitly assigning them, or by an agent claiming an application from the unassigned queue. Both paths must be implemented.

The application record has two assignment-related columns: assigned_agent_id (nullable foreign key to users) and assigned_at (nullable timestamp). When an application has no assigned agent, both of these are null. When assigned, both are populated.

For admin assignment, the admin calls the assign endpoint with the application code and the agent's user_id. The assignment service method first verifies the application exists, then verifies the agent exists and is active (is_active is true on their user record) and is a staff:agent role, then checks that the agent has accepting_cases set to true in their agent_settings record. If the application already has an assigned agent, this is a reassignment — the previous assignment is logged in the history table with an unassignment record. Then the application record is updated with the new assigned_agent_id and assigned_at. A new row is inserted into the application_assignment_history table. The status is transitioned to under_review if it is currently in received status (assignment triggers under_review). Email notifications are sent to the newly assigned agent and to the client.

For agent claiming from the unassigned queue, the agent calls the claim endpoint with the application code. The critical requirement here is that this operation must be atomic — two agents must not be able to claim the same application simultaneously. Use SQLAlchemy's with_for_update(skip_locked=True) on the SELECT query that fetches the application. This means: select the application row with a write lock, but if it is already locked by another transaction skip it rather than waiting. If the row is skipped, return an ApplicationAlreadyAssignedError. After acquiring the lock, check that assigned_agent_id is still null (to handle the case where another agent just assigned it between the lock acquisition and the check). Then perform the same assignment operations as above (update the record, write the history row, transition to under_review, send notifications).

The agent_settings table stores per-agent preferences. It has one row per agent user, created lazily (created when the agent first saves settings or when the admin creates the agent). The columns relevant to assignment are accepting_cases (boolean, default true) and daily_case_cap (nullable integer, null means no cap). Before allowing any assignment (admin or self-claim), check if the agent's accepting_cases is true. If daily_case_cap is set, count how many applications transitioned to under_review today for this agent and compare against the cap.

The unassigned queue endpoint returns all applications in received status with assigned_agent_id null, ordered by submitted_at ascending (oldest unassigned first). This endpoint is accessible to agents and admins. It returns summary data only: the application code, the service name, the submitted date, the chosen tier, and the estimated business days from the pricing tier (so agents can see which applications need the fastest attention).

## Section Twelve: Non-Negotiable Technical Rules

These rules must be followed without exception. They are not suggestions.

Never write SQL queries inside router.py files. All queries belong in repository.py.

Never write HTTP status codes, cookie logic, or response construction inside service.py files. These belong in router.py.

Never use synchronous file I/O in any async context. Always use aiofiles for reading and writing files.

Never trust the Content-Type header for file type validation. Always read the magic bytes using filetype or python-magic.

Never buffer an entire uploaded file in memory. Always stream chunks and write them to disk incrementally.

Never allow a status transition that is not in the valid transition map. Validate in the service layer and raise InvalidStatusTransitionError.

Never allow a client to see another client's application data. Enforce this with a client_id equality check in the service layer.

Never allow an agent to see another agent's application data. Enforce this with an assigned_agent_id equality check in the service layer.

Never allow a client to see internal messages. Filter them out with is_internal equals false in the repository query when called from a client-facing service method.

Never access a SQLAlchemy relationship attribute on an ORM object without having eagerly loaded it with selectinload() or joinedload() in the query. Accessing an unloaded relationship in async context will either fail or trigger a lazy load that blocks the event loop.

Never commit the session inside a repository method. Repositories call flush() only. Services call commit() after all repository operations are complete.

Never enqueue a notification email before the database transaction commits. Always enqueue after commit.

Every new endpoint must have rate limiting applied via the rate_limit dependency.

Every response must use the success_response() envelope function from core/api.py.

Every error must be raised as an AppError subclass so it is caught by the global exception handler and returned as an error_response() envelope.

Type annotations must be complete and accurate on every function parameter and return value throughout all new files.
